# repo-share

A small CLI for vendoring source files from a canonical local repo into other repos under `third_party/`, with content hashes for drift detection and read-only protection against accidental edits.

Use this when you want to share code between sibling repos on the same machine without the friction of submodules, npm publishing, or file: dependencies. The shared files are committed snapshots — there are no install-time fetches and no runtime resolution.

## Why

- **No submodules.** Submodules are awkward in monorepo-adjacent workflows and require remote access.
- **No npm publish step.** You can iterate on the canonical repo and re-sync consumers without a release cycle.
- **No file: deps.** Vendored files are committed plain text, reviewable in diffs.
- **Hash-verified.** Every copy stores a content hash; `check --locked` rejects drift without needing the canonical repo (useful in CI).
- **Read-only by default.** Copies are marked read-only on disk so you don't edit them by accident.

## Install

```bash
npm install -g repo-share
# or, from source
git clone https://github.com/sshkeda/repo-share && cd repo-share && npm link
```

## Usage

```text
repo-share add <name> --from <canonical-repo> --to <target-path> [--include a,b] [--exclude a,b]
repo-share sync [name]
repo-share check [--locked] [name]
repo-share protect [name]
repo-share list
repo-share diff [name]
```

- `add` registers a new share in `.repo-share.json` and copies files from the canonical repo into `<target-path>`.
- `sync` re-copies from the canonical repo (the canonical worktree must be clean unless you pass `--locked`).
- `check` recomputes content hashes against `.repo-share-copy.json`. With `--locked` it can run without the canonical repo present (CI mode).
- `protect` re-applies read-only permissions on copied files (run after fresh checkouts that lose `chmod`).
- `list` shows every configured share.
- `diff` shows differences between canonical source and the vendored copy.

Default excludes: `.git`, `AGENTS.md`, `node_modules`, `dist`, `coverage`, `.turbo`, `.DS_Store`, `*.tsbuildinfo`.

## Manifests

Each consumer repo owns two manifest files:

- `.repo-share.json` at the repo root — declares every share: name, canonical source repo path, target path, include/exclude lists.
- `.repo-share-copy.json` inside each target directory — machine-written metadata: source commit SHA + per-file SHA-256 hashes.

Both are committed. `.repo-share-copy.json` is regenerated by `sync` and verified by `check`.

## Agent workflow

`skills/third-party/SKILL.md` ships with this repo so AI agents working in a consumer repo know:

1. Never hand-edit files under `third_party/` or paths listed in `.repo-share.json`.
2. Treat those files as read-only evidence.
3. To change shared code: edit the canonical repo, commit it, then `repo-share sync <name>` in the consumer.
4. Use `repo-share check --locked` to verify snapshots in CI or after fresh checkouts.
